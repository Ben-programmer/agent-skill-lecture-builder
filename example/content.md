# 把講義變成網站：用 Agent Skill 建立可維護的 AI 工作流
> 這堂課想解決的，不是「怎麼叫 AI 生出一頁漂亮的 HTML」而已；真正的重點是，如何把一次性的 Vibe Coding 成果，整理成未來還能持續修改、維護、重複使用的工作流。

## 為什麼我不想再做一次性講義

### 🎤 這份講義是怎麼開始的？
- 我原本只是想準備一場分享，主題是怎麼把 AI 導入工程師日常開發工作流。
- 準備過程中，我突然想到：既然內容本身就在講 AI 寫程式，那不如乾脆讓講義本身也用 AI 協作完成。
- 於是我沒有先做投影片，而是嘗試做一個可在電腦、平板、手機瀏覽，甚至能匯出 PDF 的講義網站。

### ✨ 第一版其實已經很好看
- 只花一個晚上，模板雛形就做出來了，視覺效果和操作體驗都已經相當不錯。
- 如果目標只是「快速做出一份可以展示的講義」，那麼讓 AI 直接產生 HTML，其實很夠用。
- 問題在於，一旦你開始在意細節，調整成本就會瞬間浮現。

### ⚠️ 真正的痛點不是生成，而是修改
[tags]
- [orange] 結果有隨機性：每次生成版面與文案都可能不一樣
- [purple] 細節難微調：和 AI 對話修正不一定能精準命中需求
- [blue] 直接改 HTML 很痛苦：內容、樣式、結構全部混在一起
- [green] 缺乏可重複使用性：下次做新主題又得從頭來一次
[/tags]

> **Vibe Coding 的速度很快，但用完就丟的東西，永遠不會變成你的。**
> 如果後續很難維護，那麼初始生成再快，也只是把時間成本延後支付。

# 從零開始：先在終端機安裝 Codex，讓 AI 幫你生成第一版網頁
> 想讓 AI 有明確的工作空間，第一步不是寫提示詞，而是先把執行環境準備好。只要工作區乾淨、工具安裝正確，後面的協作體驗就會順很多。

## 給 AI 一個可工作的專案資料夾

### 📂 為什麼要先開一個空資料夾？
- AI Agent 需要一個明確的工作區，才知道檔案要放哪裡、改哪些內容。
- 不管你使用 VSCode、Cursor、Antigravity，核心概念都一樣：先開一個專案，再讓 AI 在裡面動手。
- 這次我用 Antigravity，只是因為方便檢視檔案變更，不是因為它內建的 AI 功能。

## 確認 Node.js 與安裝方式

### 🧪 檢查 Node.js 版本
```prompt [label="確認 Node.js"]
node -v
```

- 如果終端機能正常顯示版本號，代表環境已具備基礎條件。
- 如果找不到指令，就要先安裝 Node.js，否則後面無法執行腳本與工具。
- 我會建議透過 NVM 安裝，因為之後切換版本會省事很多。

### 📦 使用 NVM 安裝 Node.js
```prompt [label="安裝 NVM（macOS）"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

```prompt [label="安裝 Node.js LTS"]
nvm install --lts
node -v
```

- Windows 使用者則建議安裝 `nvm-windows` 的執行檔版本。
- NVM 安裝完成後，記得開一個新的終端機再驗證 `nvm -v`。
- 這樣之後遇到工具要求不同 Node 版本時，也比較好調整。

## 安裝 Codex 並完成登入

### 🤖 終端機全域安裝
```prompt [label="安裝 Codex"]
npm i -g @openai/codex
```

### 🔐 Codex 提供的三種登入方式
[flow]
1. 使用 ChatGPT 帳號登入 — 最直覺，也是最推薦的方式
2. 使用裝置代碼登入 — 適合沒有瀏覽器介面的 VM 或遠端機器
3. 使用 OpenAI API Key — 依用量計費，適合已有 API 管理需求的情境
[/flow]

- 登入完成後，Codex 會先詢問你是否信任這個資料夾，選擇信任後才能繼續操作。
- 後續出現的輸入框，就可以把它當成 AI 對話視窗來使用。

# 先讓 AI 生出第一版，再看清楚問題出在哪裡
> 不需要一開始就追求完美流程。先讓 AI 生成一次，親自感受它哪裡好、哪裡難改，之後才知道模板該怎麼設計。

## 在終端機請 AI 生成講義網站

### 📝 第一段核心提示詞
```prompt [label="生成講義網頁"]
請扮演熟悉 AI 主題授課的講師，以「Agent Skills」為主題設計講義
講義用純 html 網頁（CSS/JS 都放裡面）呈現
日系風格的設計感，部分文字與背景可以搭配適當的漸層
開頭會根據內容呈現標題，然後下面有發人深省的標語
會根據不同層級與意圖呈現對應的區塊（ex: 大小標題、陳述、列點、流程）
最後放上社群連結鼓勵大家訂閱（FB/YouTube/Medium）
側邊欄會有對應的目錄，方便跳轉到對應位置
```

### 🎯 這段提示詞為什麼有效？
[summary]
- 🎯 **目標明確** | 限定輸出是單一 HTML，部署與預覽都更簡單
- 🎨 **風格先講清楚** | 日系、漸層、標語感，讓視覺方向更一致
- 🧱 **結構分層描述** | 大標、小標、列點、流程、目錄、社群連結都先定義好
[/summary]

### 👀 第一版結果通常會發生什麼事？
- 完成度往往比預期高，甚至已經可以直接拿去展示。
- 但當你開始調整段落順序、語氣、卡片層次或特定區塊時，就會感受到 AI 對話式微調的不穩定。
- 如果改動很多，直接碰 HTML 的成本又會變得很高。

> **生成速度從來不是唯一答案**
> 真正影響長期效率的，是你能不能把內容抽離成穩定、好維護、能反覆重建的格式。

# 逆向拆解成模板：把 HTML 變成 Markdown，再封裝成 Skill
> 與其每次叫 AI 從零生成結果，不如先請它幫你建立一個可編輯的模板。內容變成 Markdown，輸出交給腳本，這樣你才真正拿回掌控權。

## 第一步：把剛生成的 HTML 逆向整理成 Markdown 規格

### 🔄 對 AI 下的第二段提示詞
```prompt [label="把 HTML 轉成模板工作流"]
我希望剛剛生成出來的網頁，是可以透過 Markdown 格式編輯內容的
請幫我將網頁結構拆解（列點、步驟、圖片、表格、註解）、分層次（大標、中標、小標），並以常見的 Markdown 語法來設計規則
最後建立一個腳本，我只要編輯 Markdown 文件，網頁的 HTML 就會及時改變
操作方式寫在 README.md 文件中，並建立 package.json
```

### 🧩 這段提示詞實際上在做哪四件事？
[flow]
1. 把最終輸出的 HTML，回推成可維護的內容結構
2. 用 Markdown 定義內容規則，讓講義不再綁死在 HTML
3. 建立 build script，讓內容調整能立即反映到頁面
4. 補 README 與 package.json，讓整個流程可被重複操作
[/flow]

- 這不是單純把內容轉成 Markdown 而已，而是在建立一套長期可維護的製作流程。
- 當 Markdown 成為輸入、HTML 成為輸出之後，AI 與人類都比較容易共同編修同一份內容。

## 第二步：把流程封裝成 Agent Skill

### 🛠️ 建立 Skill 的需求描述
```prompt [label="建立 course-page Skill"]
根據下面需求在「.agents」資料夾建立 Skill：
- 參考模板：根據剛剛腳本設計的規格，建立明確的 Markdown 使用規範
- 觸發時機：使用者提供初步想法想建立網頁時，會觸發這個 Skill
- 執行順序：這個 Skill 會先深入研究輸入的主題，再根據規範設計對應的 Markdown 文件；最後把 Markdown 文件透過腳本生成 html
- 特別提醒：預計會有多個主題，所以觸發時，請先建立新的資料夾，裡面存放新生成的 Markdown 與 HTML 檔案；並且建立獨立的 scripts 與 references 資料夾，確保可單獨執行
```

### 🧠 Skill 幫你解決的是什麼？
- 它把原本零散的對話流程，整理成固定會被觸發的工作規範。
- 未來只要丟主題或講稿，AI 就知道要先建資料夾、整理內容、套用格式、建置頁面。
- 這讓每次產生新講義時，不需要重新說明同一套規則。

### 📁 一個可重複使用的 Skill 通常包含哪些東西？
[tags]
- [blue] `SKILL.md`：定義用途、觸發時機、執行步驟
- [green] `reference/`：放格式規範與內容範例
- [purple] `scripts/`：放 build、dev、產圖等執行腳本
- [orange] 課程資料夾：存放每個主題自己的 Markdown 與輸出頁面
[/tags]

> **模板的價值，在於把隨機輸出變成穩定工作流**
> 只要規則清楚，AI 每次做的就不只是「幫你生一份東西」，而是跟著團隊定義好的方式產出成果。

# 從工具崇拜回到工作流設計：真正該培養的是判斷力
> 我一直更換不同的 AI 工具，不是因為每次都想追最新，而是因為我想訓練自己在不同工具之間切換、比較、組合的能力。

## 不要把焦點放在「哪個工具最強」

### 🔁 為什麼我要刻意使用多個 AI 工具？
- 因為只要某個 AI 被捧上天，它很快就可能被下一個工具取代。
- 工具差異存在，但通常沒有行銷文案說得那麼巨大。
- 真正能累積的能力，是你是否知道怎麼描述需求、怎麼拆流程、怎麼驗證成果。

### 🧭 這堂課真正想帶走的觀念
[summary]
- 🧱 **先建模板，再追求美觀** | 可維護性比一次生成的驚豔感更重要
- 🤝 **讓 AI 當協作者，不是魔法箱** | 規則、格式、腳本都要設計清楚
- 🔄 **把流程產品化** | 當工作流能重複執行，成果才會累積成資產
- 🧠 **培養跨工具判斷力** | 重點不是綁定某個平台，而是理解何時用、怎麼用
[/summary]

## 如果你想在編輯器裡操作 Codex

### 🧩 外掛只是入口，不是核心
- 你可以在支援的編輯器中安裝 Codex 外掛，用對話窗和 AI 互動。
- 但無論是終端機還是編輯器，真正重要的都不是介面，而是你背後設計的工作流。
- 同樣一套 Skill、同樣一份 Markdown 規格，可以在不同入口重複使用。

## 最後的提醒

### 🚀 別讓成果停在「看起來可以用」
- 如果只是想快速做一份展示頁面，Vibe Coding 已經很夠用了。
- 但如果你希望它未來還能迭代、交接、擴充，就要往模板、Skill、腳本與規格這個方向設計。
- 真正值錢的，不是某個單一工具，而是你怎麼把工具串成一條能持續產出成果的路徑。

[summary]
- 🌐 **先生成第一版** | 用 AI 快速確認想像中的講義網站長什麼樣子
- 📝 **再抽出 Markdown 規格** | 讓內容可編輯、可維護、可交接
- 🛠️ **最後封裝成 Agent Skill** | 把一次性的流程變成往後都能重複執行的能力
[/summary]
